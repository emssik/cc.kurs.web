# Lekcja 02.09 — MCP Part 1 - Podstawy i Architektura (Research dla podcastu)

## Trudne tematy techniczne

- **MCP (Model Context Protocol) jako standard komunikacji** — samo pojęcie "protokołu" jest abstrakcyjne dla osób nietechnicznych. Trudno zrozumieć czym różni się "protokół" od "programu" czy "aplikacji".

- **Problem N×M vs N+M (eksplozja integracji)** — wymaga myślenia matematycznego/systemowego. Dla osoby nietechnicznej "50 integracji vs 15 komponentów" brzmi jak abstrakcja bez kontekstu.

- **Różnica między RAG, MCP i Function Calling** — trzy koncepty techniczne porównywane jednocześnie. Dla początkującego to trzy nowe pojęcia naraz, z których każde wymaga osobnego zrozumienia zanim zacznie się je porównywać.

- **Architektura Host / Client / Server** — pojęcia "klient" i "serwer" są używane w IT od lat, ale dla osoby spoza branży to zupełnie nowy model mentalny. Dodanie "Hosta" jako trzeciej warstwy komplikuje sprawę.

- **Trzy prymitywy: Resources, Tools, Prompts** — słowo "prymityw" jest samo w sobie techniczne. Rozróżnienie między "danymi do odczytu" (Resources) a "akcjami" (Tools) wymaga zrozumienia jak systemy komputerowe operują na danych.

- **Function Calling jako mechanizm wykonawczy** — pojęcie "wywołania funkcji" jest naturalne dla programistów, ale dla marketera czy HR-owca wymaga wyjaśnienia czym w ogóle jest "funkcja" w kontekście programowania.

- **Transporty: stdio, HTTP/SSE, JSON-RPC** — lekcja celowo je pomija, ale wymienia je mimochodem. Dla osoby nietechnicznej nawet sama wzmianka może budzić niepokój ("czy muszę to rozumieć?").

- **Tokeny, uprawnienia, principle of least privilege** — koncepty bezpieczeństwa IT. Osoba nietechniczna może nie rozumieć czym jest "token", dlaczego "admin" to za dużo uprawnień i co oznacza "separacja środowisk dev/prod".

- **Serwer lokalny vs zdalny** — rozróżnienie między programem działającym na własnym komputerze a usługą w chmurze wymaga podstawowego zrozumienia infrastruktury IT.

- **API (Application Programming Interface)** — pojawia się wielokrotnie (GitHub API, API bazy danych). Dla osoby nietechnicznej to kluczowe pojęcie, bez którego reszta nie ma sensu, a samo w sobie jest abstrakcyjne.

## Prompt dla NotebookLM Deeper Search

Przygotuj podcast wprowadzający dla osób nietechnicznych (marketerzy, HR-owcy, pisarze, project managerowie) na temat Model Context Protocol (MCP) — nowego standardu łączenia narzędzi AI z zewnętrznymi usługami.

Kontekst: Słuchacze uczestniczą w kursie Claude Code i właśnie dostali lekcję o MCP. Lekcja używa analogii (MCP jako "port USB-C dla AI", architektura jako "restauracja z kelnerem i kuchnią"), ale wiele konceptów technicznych wymaga dodatkowego wyjaśnienia.

Tematy do omówienia w podkaście:

1. **Co to jest protokół komunikacji** — wyjaśnij na codziennych przykładach (np. język, zasady ruchu drogowego), zanim przejdziesz do MCP. Słuchacz musi zrozumieć samo słowo "protokół".

2. **Problem eksplozji integracji (N×M vs N+M)** — użyj analogii z życia codziennego. Np. wyobraź sobie, że każdy producent gniazdek elektrycznych wymaga osobnej wtyczki od każdego producenta urządzeń. Pokaż dlaczego jeden standard oszczędza pracę.

3. **Czym jest API** — to pojęcie pojawia się w lekcji wielokrotnie (GitHub API, baza danych). Wyjaśnij je tak, żeby osoba, która nigdy nie programowała, mogła je zrozumieć. Użyj analogii: API to jak menu w restauracji — lista rzeczy, które możesz zamówić, z opisem co dostaniesz.

4. **RAG vs MCP vs Function Calling** — lekcja porównuje te trzy koncepty. W podkaście rozłóż je na czynniki pierwsze: RAG to czytanie encyklopedii (dane statyczne), MCP to rozmowa telefoniczna (dane na żywo + akcje), Function Calling to mechanizm techniczny pod spodem. Wyjaśnij dlaczego użytkownik nie musi rozumieć Function Calling, ale powinien rozumieć różnicę między RAG a MCP.

5. **Architektura Host-Client-Server** — lekcja używa analogii restauracji (Host = budynek, Client = kelner, Server = kuchnia). Rozwiń tę analogię, ale też wyjaśnij dlaczego ta architektura ma znaczenie praktyczne: np. możesz mieć jednocześnie "kelnera do GitHub" i "kelnera do Slacka", a Claude koordynuje ich pracę.

6. **Resources, Tools, Prompts — trzy rodzaje "rzeczy" w MCP** — wyjaśnij różnicę między danymi do czytania (Resources), przyciskami do klikania (Tools) i gotowymi przepisami (Prompts). Użyj przykładów z życia: Resources to jak tablica ogłoszeń, Tools to jak przyciski w windzie, Prompts to jak checklisty przed lotem.

7. **Bezpieczeństwo dla nietechnicznych** — lekcja mówi o tokenach, uprawnieniach, principle of least privilege. Wyjaśnij te koncepty prostym językiem: token to jak klucz do mieszkania (daj komuś klucz tylko do pokoju, który potrzebuje, nie do całego budynku). Separacja dev/prod to jak ćwiczenie na sucho przed prawdziwym występem.

Styl podkastu: rozmowa dwóch osób, lekki ton, dużo analogii z codziennego życia. Unikaj żargonu technicznego — a jeśli musisz go użyć, natychmiast wyjaśnij. Cel: po wysłuchaniu podkastu słuchacz powinien wrócić do lekcji i rozumieć ją bez problemu.
