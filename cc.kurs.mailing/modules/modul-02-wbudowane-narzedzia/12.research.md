# Lekcja 02.12 — MCP Part 4: Bezpieczenstwo i Katalog Serwerow (Research dla podcastu)

## Trudne tematy techniczne

- **Prompt Injection (Wstrzykniecie Polecen)** — Koncept, ze model AI nie odroznia danych zewnetrznych od instrukcji uzytkownika, jest nieintuicyjny. Wymaga zrozumienia, czym jest "kontekst" modelu i dlaczego komentarz HTML w pliku README moze przejac kontrole nad sesja.

- **Tool Poisoning (Zatruwanie Narzedzi)** — Idea, ze opis narzedzia (tool description) moze zawierac ukryte instrukcje niewidoczne dla uzytkownika, ale widoczne dla modelu. Wymaga zrozumienia, jak model "czyta" definicje narzedzi i dlaczego slepemu zaufaniu do npm packages nie mozna ufac.

- **Cross-Repository Data Theft (Kradziez danych miedzy repozytoriami)** — Mechanizm, w ktorym publiczne repozytorium moze wykrasc dane z prywatnego, jesli uzywa sie tego samego tokena. Wymaga zrozumienia koncepcji tokenow dostepu, scopeow uprawnien i separacji kontekstow.

- **Defense in Depth (Obrona wielowarstwowa)** — Strategia laczaca sandbox, least privilege, hooki i separacje tokenow. Dla osoby nietechnicznej trudne jest zrozumienie, dlaczego jedna warstwa obrony nie wystarczy i jak te warstwy wspoldzialaja.

- **PreToolUse Hook (Hak pre-narzedziowy)** — Skrypt bash uruchamiany automatycznie przed kazdym wywolaniem narzedzia. Wymaga zrozumienia koncepcji hookow, JSON-a, pipe'ow w terminalu i logiki warunkowej w skryptach.

- **Sandbox Mode i Least Privilege** — Pojecia z bezpieczenstwa informatycznego: ograniczanie uprawnien do minimum, izolacja procesow. Dla osoby nietechnicznej niejasne moze byc, czym jest "sandbox" i dlaczego blokowanie uprawnien to cos dobrego.

- **OAuth vs Personal Access Token (PAT) vs Fine-Grained Token** — Roznice miedzy metodami uwierzytelniania. Trudne: czym jest token, czym sie rozni OAuth od klucza API, co to sa "scopes" i dlaczego wazna jest rotacja tokenow.

- **Docker Isolation** — Uruchomienie serwera MCP w kontenerze Docker bez dostepu do sieci. Wymaga zrozumienia koncepcji konteneryzacji, izolacji procesow i flag typu `--network=none`.

- **CVE (Common Vulnerabilities and Exposures)** — System identyfikacji luk bezpieczenstwa. Trudne: co to jest numer CVE, co oznacza CVSS 9.4, dlaczego wersja oprogramowania ma znaczenie.

- **MCP Inspector i jego luki (CVE-2025-49596)** — Narzedzie do debugowania serwerow MCP, ktore samo mialo krytyczna luke (RCE przez CSRF + DNS rebinding). Wymaga zrozumienia wielu koncepcji naraz: GUI testowe, zdalne wykonanie kodu, ataki sieciowe.

- **JSON-RPC, stdio, HTTP/SSE — transporty MCP** — Protokoly komunikacji miedzy Claude Code a serwerami MCP. Dla osoby nietechnicznej niejasne: czym jest stdin/stdout, czym rozni sie komunikacja lokalna od zdalnej, co to streaming.

- **Managed Settings i hierarchia konfiguracji** — Koncept, ze konfiguracja ma warstwy (user -> project -> local -> managed) z roznymi priorytetami. Trudne: dlaczego IT admin moze nadpisac ustawienia uzytkownika i jak to dziala technicznie.

- **Debugowanie serwerow MCP (--mcp-debug, logi, decision tree)** — Czytanie logow, kodow bledow (exit code), brakujacych zaleznosci. Wymaga bazowej znajomosci terminala i rozumienia, czym sa dependencje, PATH, uprawnienia plikow.

- **Checklist oceny jakosci serwera MCP (7 punktow)** — Ocena kodu zrodlowego, zaleznosci, licencji, testow. Trudne dla osoby, ktora nigdy nie przegladala repozytorium na GitHubie ani nie analizowala package.json.

## Prompt dla NotebookLM Deeper Search

Przygotowuje podcast wprowadzajacy do trudnych koncepcji technicznych z lekcji o bezpieczenstwie MCP (Model Context Protocol) w Claude Code. Odbiorcy to osoby poczatkujace, nietechniczne — marketerzy, HR-owcy, pisarze — ktorzy ucza sie korzystac z AI w pracy.

Podcast powinien wyjasnic nastepujace tematy prostym jezykiem, z uzyciem analogii z zycia codziennego:

1. **Czym jest atak Prompt Injection i dlaczego AI daje sie nabrac?** Wytlumacz, ze model AI traktuje wszystko w kontekscie jako rownowazne instrukcje — nie odroznia "polecenia od uzytkownika" od "tekstu z pliku README". Uzyj analogii: to jak asystent, ktory czyta na glos kazda kartke, ktora dostanie — nie sprawdza, kto ja napisal. Opowiedz o realnym przypadku z Docker Blog, gdzie Claude wykradl klucze API bo README z publicznego repo zawieralo ukryte instrukcje.

2. **Czym jest Tool Poisoning i jak zlosliwe narzedzie moze przejac kontrole?** Wytlumacz, ze serwery MCP to jak aplikacje — moga miec ukryte funkcje w swoich opisach, ktorych uzytkownik nie widzi, ale model AI tak. Analogia: to jak pobranie aplikacji, ktora mowi "notatnik", ale w tle wysyla Twoje pliki. Wspomnij o CVE-2025-6514 (mcp-remote) jako przykladzie z zycia.

3. **Co to jest Cross-Repository Data Theft?** Wytlumacz najprosciej: jesli ten sam klucz otwiera drzwi do prywatnego i publicznego pokoju, to ktos w publicznym pokoju moze wyslac AI po dokumenty z prywatnego. Opisz scenariusz fintech z lekcji.

4. **Jak sie bronic — Defense in Depth prostym jezykiem.** Wytlumacz strategie wielowarstwowej obrony: sandbox (pokojowa klatka), least privilege (daj klucze tylko do potrzebnych pokojow), osobne tokeny (rozne klucze do roznych drzwi), hooki (straznik przy drzwiach sprawdzajacy kazda przesylke). Podkresl, ze zadna pojedyncza warstwa nie wystarczy.

5. **Czym sa tokeny dostepu (PAT, Fine-Grained, OAuth) i dlaczego to wazne?** Wytlumacz roznice: PAT to klucz-wytrych (otwiera wszystko), Fine-Grained Token to klucz do jednego pokoju, OAuth to system, gdzie prosisz o tymczasowy dostep do konkretnej rzeczy. Wspomnij statystyke: 53% implementacji MCP uzywa starych, nierotowanych tokenow.

6. **Jak ocenic, czy serwer MCP jest bezpieczny — praktyczny checklist.** Przejdz przez 7 kryteriow z lekcji prostym jezykiem: kto to napisal, ile osob tego uzywa, czy jest dokumentacja, czy kod wyglada dobrze, czy sa testy, ile ma zaleznosci, jaka licencja. Analogia: to jak sprawdzanie restauracji przed rezerwacja — opinie, menu, higiena.

7. **Debugowanie dla poczatkujacych — co robic, gdy cos nie dziala.** Wytlumacz, ze `--mcp-debug` to jak "tryb diagnostyczny" samochodu — pokazuje dokladnie, co sie zepsulo. Przejdz przez najczestsze problemy: brak zaleznosci, timeout, brak uprawnien.

Ton podcastu: przyjazny, bez protekcjonalnosci, z konkretnymi przykladami z lekcji. Unikaj zargonu bez wyjasnienia. Kazdy temat zilustruj analogia do czegos, co osoba nietechniczna zna z codziennego zycia.
